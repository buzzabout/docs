The buzzabout API uses **bearer tokens** in the `Authorization` header.
There are no cookies, OAuth flows, or session tokens ‚Äî just a key.

## Obtaining a key

Open **Settings ‚Üí API keys** in the web app. Two kinds:

| Kind          | Use it for                                                       |
| ------------- | ---------------------------------------------------------------- |
| **Standard**  | Personal scripts, internal jobs, full workspace access.          |
| **Scoped**    | Production integrations. Pick the smallest set of scopes needed. |

Keys are shown **once** at creation. We hash them at rest, so we can't
re-display the secret if you lose it.

## Sending the key

```http
GET /v1/topics HTTP/1.1
Host: api.buzzabout.ai
Authorization: Bearer YOUR_API_KEY
```

```bash title="cURL"
curl https://api.buzzabout.ai/v1/topics \
  -H "Authorization: Bearer $BUZZABOUT_KEY"
```

## Verifying a key

Hit `/v1/me`. It returns the workspace, the scopes attached to the
key, and the rate-limit headroom ‚Äî useful as a smoke test.

```bash
curl https://api.buzzabout.ai/v1/me \
  -H "Authorization: Bearer $BUZZABOUT_KEY"
```

```json title="200 response"
{
  "workspace": { "id": "ws_01H...", "name": "Acme Marketing" },
  "key": {
    "id": "key_01H...",
    "scopes": ["topics:read", "analyses:read", "mentions:read"],
    "created_at": "2026-04-12T11:01:00Z"
  }
}
```

## Scopes

| Scope             | Read endpoints                          | Write endpoints                              |
| ----------------- | --------------------------------------- | -------------------------------------------- |
| `topics:read`     | `GET /v1/topics`, `GET /v1/topics/:id`  | ‚Äî                                            |
| `topics:write`    | ‚Äî                                       | `POST /v1/topics`, `PATCH /v1/topics/:id`    |
| `analyses:read`   | `GET /v1/analyses/:id`                  | ‚Äî                                            |
| `analyses:write`  | ‚Äî                                       | `POST /v1/analyses`                          |
| `mentions:read`   | `GET /v1/mentions`                      | ‚Äî                                            |
| `webhooks:write`  | ‚Äî                                       | `POST /v1/webhooks`, `DELETE /v1/webhooks/:id` |

A key without scopes acts as `*` (everything in the workspace). Treat
unscoped keys like database root ‚Äî keep them off shared machines.

## Rotation

There's no built-in expiration. You rotate by:

1. Creating a new key with the same scopes.
2. Updating your environment / secret store.
3. Revoking the old key from **Settings ‚Üí API keys**.

Revoked keys return `401 unauthenticated` immediately on the next call ‚Äî
there's no grace period.

<Callout type="warn" title="Don't commit keys to Git">
  Even short-lived test keys leak into PR diffs. Use a `.env` file
  ignored by `.gitignore` or a secret manager.
</Callout>
