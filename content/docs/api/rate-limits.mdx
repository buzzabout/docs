The API enforces two limits per key:

| Limit                | Window      | Default               |
| -------------------- | ----------- | --------------------- |
| Requests             | rolling 60s | 60 requests / minute  |
| Concurrent analyses  | live        | 5 in-flight           |

Larger plans get higher numbers — see the
[pricing page](https://buzzabout.ai/Pricing.html). If you need a custom
limit for a specific integration, mail support.

## What you get back on every request

Every response (success or failure) includes three headers so you can
track headroom without polling:

```http
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 47
X-RateLimit-Reset: 1717091400
```

`X-RateLimit-Reset` is a Unix timestamp — the wall-clock second at
which the bucket fully refills.

## What `429` looks like

```json title="429 body"
{
  "error": {
    "type": "rate_limited",
    "message": "60 requests per minute exceeded.",
    "retry_after_ms": 8500
  }
}
```

The response also sets a `Retry-After` header (seconds, integer). Trust
it — sleep at least that long before retrying.

## Recommended client behaviour

<Callout type="info">
  The official SDKs already do this. If you're calling the REST API
  directly, copy the pattern.
</Callout>

```ts title="retry.ts"
async function call(url: string, init: RequestInit, attempt = 0): Promise<Response> {
  const res = await fetch(url, init);
  if (res.status !== 429 || attempt >= 5) return res;

  const retry = Number(res.headers.get('retry-after') ?? 1);
  // jittered exponential back-off, capped at 60s
  const delay = Math.min(60_000, (retry * 1000) * 2 ** attempt + Math.random() * 250);
  await new Promise(r => setTimeout(r, delay));
  return call(url, init, attempt + 1);
}
```

## Concurrency limits

`POST /v1/analyses` can return `429` even when the rolling-60s budget
isn't spent — that's the concurrent-analyses cap. The body's
`error.type` distinguishes:

| `error.type`       | Meaning                                                   |
| ------------------ | --------------------------------------------------------- |
| `rate_limited`     | You hit the per-minute request cap.                       |
| `concurrency_limit`| You have too many analyses running at once.               |

For `concurrency_limit`, polling more aggressively won't help — wait
for one of your running analyses to finish (poll
`GET /v1/analyses/:id`).
